# Datathon 2026 Challenge

## Hybrid Listing Search and Relevance Ranking for Robin

## Overview

In this challenge, you will build a search and ranking system for real-estate listings.

Your task is to take a natural-language user query and return the listings that:

- satisfy the hard constraints in the query
- are sorted by soft preferences and overall relevance

This challenge is inspired by a real-world listing search problem.

You do not need to reproduce any production stack. The core challenge is search quality.

## The Problem

Real user queries mix two very different kinds of intent:

- hard criteria that must be respected
- soft preferences that should influence ranking

Example queries:

- "3-room bright apartment in Zurich under 2800 CHF with balcony, close to public transport"
- "Bright family-friendly flat in Winterthur, not too expensive, ideally with parking"
- "Modern studio in Geneva for June move-in, quiet area, nice views if possible"
- "Looking for a affordable student accomondation, max half an hour door to door to ETH Zurich by public transport, i like modern kitchens."

In all four examples, some parts are strict filters and some parts are softer ranking hints.

Your system should separate these two layers and combine them effectively.

## Your Task

Build a system that maps:

`user query -> hard filters -> candidate listings -> relevance ranking`

At interaction step `t0`, assume that you only have the current user query.

### Core task

Given a user query:

1. Identify the hard criteria
2. Retrieve listings that satisfy those criteria
3. Interpret the remaining soft preference signal
4. Rank the valid listings by relevance

Your result should ideally support:

- a relevance-sorted list of listings
- if available map-ready highlights using coordinates
- a short explanation of why the top results rank highly

### Bonus task

Use longer-term user interaction or preference history to improve ranking.

Examples of additional signals:

- inferred preference documents
- past searches
- summaries
- favorites
- clicks
- detail views
- dwell time
- hides or skips
- historical query patterns
- an explicit or inferred user preference graph

Bonus question:

How should ranking change when the same user repeatedly prefers certain locations, price bands, listing styles, image types, or building features? How can you incorporate it.

## Directions You Can Explore

You are free to choose different solution styles as long as the core task is addressed and the final outcome is strong.

Possible directions include:

- a retrieval and ranking pipeline using listing text only
- a retrieval and ranking pipeline using text and images
- an LLM-assisted query understanding layer
- a hybrid lexical and embedding-based search system
- a multimodal ranking system using text and images
- a data enrichment pipeline using external data sources such as SBB data, store or amenity data, neighborhood context, or public transport data
- commute-time-aware ranking
- geospatial analysis and proximity features, for example via geocoding or location APIs such as Nominatim
- querying multiple data sources with retrieval, grounding, embeddings, or external APIs
- Multi turn conversation, clarification questions generated by model to avoid ambiguity
- labeling or generating more training or evaluation data
- a lightweight demo UI
- integration into a ChatGPT / MCP / Apps SDK prototype that exposes your search layers as tools

Everything that helps solve the core challenge is encouraged.

The core judging focus is the final ranking quality and the breadth of query types your system can handle well.

Evaluation style queries we may judge on (dont worry if you cant fulfill all, you can also either go deep or broad):

- Ich suche eine 2,5-Zimmer-Wohnung in Zürich Kreis 4 oder 5, maximal 2’800 CHF, mit Balkon und Waschmaschine in der Wohnung. Wichtig sind mir eine moderne Küche und ein lebendiges, aber nicht zu lautes Quartier.
- Ich möchte eine Wohnung in der Nähe meines Arbeitsplatzes in Zug, mit maximal 25 Minuten Pendelzeit mit dem ÖV, mindestens 65 m² und mindestens 2 Zimmern. Die Wohnung sollte hell, ruhig und gut geschnitten sein.
- Wir sind eine Familie mit einem Kind und suchen in Winterthur eine Wohnung mit mindestens 3,5 Zimmern, ab 80 m² und Miete unter 3’200 CHF. Uns sind gute Schulen, eine kinderfreundliche Umgebung und viel Tageslicht wichtig.
- Ich suche ein Studio oder 1,5-Zimmer-Apartment in Lausanne, maximal 1’900 CHF, möglichst möbliert und mit guter Anbindung an die EPFL. Es sollte modern, sauber und möglichst in Seenähe sein.
- Ich suche eine 4-Zimmer-Wohnung in Basel, unter 3’500 CHF, mit mindestens 2 Badezimmern und Parkplatz. Schön wäre eine ruhige Wohnlage, grüne Umgebung und ein gepflegtes Gebäude.

Some hints:

ETH commute + lifestyle mix
“Bright 2-room apartment within 20 minutes of ETH Zentrum by public transport, under CHF 2,200, ideally in a lively neighbourhood with cafés”
→ Hard: rooms=2, price≤2200, commute constraint. Soft: bright, lively, café density. Tests transport enrichment (SBB/ZVV) + POI data.

PhD-style long-term, quiet
“Quiet 1.5-room flat near ETH Hönggerberg, furnished if possible, move-in March, max CHF 1,800”
→ Hard: rooms=1.5, price≤1800, available~March, proximity to Hönggerberg campus. Soft: quiet, furnished. Tests geospatial radius around a specific campus + description parsing for “möbliert”.

Classic family conflict query
“Family-friendly 4.5-room apartment in a Zurich suburb with good S-Bahn connection to HB, garden or large balcony, near good schools, up to CHF 3,500”
→ Hard: rooms≥4.5, price≤3500, Zurich commuter belt, outdoor space. Soft: “good schools”, “good connection”. Tests POI enrichment (schools) + commute time computation.

Multimodal / image-heavy
“Modern loft-style apartment in Zurich Kreis 4 or 5 with lots of natural light and a view, 3+ rooms, around CHF 2,800”
→ Hard: Kreis 4/5, rooms≥3, price~2800. Soft: “modern”, “loft-style”, “natural light”, “view”. Forces image analysis — text alone won’t distinguish these well.

Conflicting / tradeoff query
“Cheap but central studio in Zurich, close to ETH and nightlife, doesn’t matter if small”
→ Hard: studio/1 room, Zurich. Soft + conflicting: cheap vs. central vs. near ETH vs. nightlife. Tests how the system handles contradictions and explains its ranking tradeoffs — exactly the Strategy A/B/C pattern from slide 10.

## What You Need to Build

You are free to choose the format of your solution, as long as it clearly solves the problem.

You may use the provided starter harness, adapt parts of it, or ignore it entirely and build your own solution from scratch.

Examples:

- a retrieval and reranking pipeline
- an API or service
- a notebook-based prototype
- an interactive demo
- a lightweight search UI
- a ChatGPT / MCP / Apps SDK prototype

The primary focus is search and ranking quality, not frontend polish.

## Expected Output

Your system should return listings that:

- satisfy the hard constraints in the query
- are ordered by soft relevance

Outputs should at least include:

- the final ranked results as a list of `listing_id` values with some form of scoring

It may include:

- the extracted hard filters
- the scoring or ranking logic at a high level
- brief explanations for top-ranked listings

## Solution Ideas

There is no single correct approach. Strong participant solutions may include one or more of the following:

- rule-based or model-based hard filter extraction
- structured retrieval over listing metadata
- lexical search such as BM25
- embedding-based retrieval or reranking
- LLM-based query decomposition into `must-have`, `nice-to-have`, and negative signals
- cross-encoder reranking on top candidates
- hybrid retrieval combining structured filters, keyword search, and embeddings
- multimodal ranking using both listing text and images
- enrichment features derived from external data such as commute times, shops, schools, public transport, or neighborhood signals
- geospatial and map-aware scoring
- multi-source retrieval or grounded ranking across internal and external APIs
- preference learning using favorites or interaction history
- per-query explanation generation

You may use external models or APIs, but you must clearly document them.

## Design Challenges

This problem is interesting because user language is messy and real listings are incomplete.

Examples of things your system may need to handle:

- vague terms like "bright", "modern", "family-friendly", or "good location"
- missing constraints
- conflicting preferences
- tradeoffs between price, location, size, and quality
- descriptions where soft preferences are implied rather than explicitly labeled
- images that may carry useful relevance signals

## Evaluation

Submissions will be evaluated using a mix of automatic scoring and human judgment and peer review.

### Automatic leaderboard

The ranking core may be evaluated with metrics such as:

- hard-filter precision
- constraint satisfaction rate
- ranking

Important principle:

A system that regularly violates hard constraints by more than a margin is not a strong solution, even if its soft ranking looks good.

### Jury review

The jury will look at:

- correctness and robustness
- relevance quality
- originality of the ranking approach
- practical usefulness
- quality of the demo and presentation

### Peer feedback

Peer testing also plays a role, especially for:

- perceived usefulness
- search experience
- trust in results

## Rules

- Use the provided listing dataset as the main source of truth.
- You may use external models and APIs, but you must declare them.
- Your submission should be understandable and reproducible at a high level.
- Your system should handle ambiguous or underspecified queries gracefully.
- If you build a full MCP or app experience, make sure the core ranking behavior is still easy to evaluate.

## Tips

- Start by separating strict filters from softer ranking hints.
- Do not rely on semantic similarity alone.
- Do not rely on structured filtering alone.
- Hybrid systems are often stronger than single-method systems.
- You can make your ranking behavior explainable enough that a human can understand why the top results appeared so you can debug it.
- If you attempt the bonus task, think carefully about how to use user history without overfitting to one past action, how you can inject it.

## Success Criteria

A strong submission should:

- remove listings that violate hard constraints
- rank the remaining listings in a useful order
- perform well on vague and natural user queries
- make sensible tradeoffs between multiple preferences
- enables either broad options to search for or excellent depth for some areas of expertise, or both ;)
- show thoughtful use of personalization in the bonus task

## Final Brief

Given a user query, find the listings fulfilling the hard criteria and sort them by soft criteria and overall relevance.

Bonus: improve the ranking using longer-term user interaction or a user preference graph.
