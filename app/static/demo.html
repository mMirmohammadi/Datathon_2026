How the ranking is derived (click to collapse)
  1. LLM extraction (gpt-4o-mini, strict JSON schema): the query is parsed into hard filters (city, postal code, canton, price range, rooms range, features) and soft preferences (cheap / quiet / near public transport / near schools / near park / family friendly / commute target / landmark proximity). Vague adjectives never become hard filters — "modern" is never turned into new_build.
  2. Hard-filter gate (SQL WHERE): listings that violate any hard constraint are removed — they never compete for ranking. This is strict, not scored.
  3. BM25 lexical relevance (SQLite FTS5, always on): the bm25_keywords the LLM extracted are matched against title × 3, description × 1, street × 0.5, city × 0.5. The candidate pool arrives pre-ordered by BM25.
  4. Visual SigLIP-2 Giant (cosine on 1536-d image embeddings, 70 548 photos): each candidate's best image is scored against the query text. Max cosine per listing; listings without images are omitted from this channel, not penalised.
  5. Semantic Arctic-Embed-L v2 (cosine on 1024-d description embeddings, 25 546 listings): the query is re-written with the query: prefix and scored against each description vector.
  6. Soft-preference rankings (one per activated key): cheap/premium sort by canton×rooms price delta (dropping price_plausibility=suspect outliers), quiet by distance to motorway + primary road, near_public_transport by a composite of stop distance and line count, commute_target by precomputed commute minutes to the HB, near_schools/supermarket/park by POI counts, near_landmark by per-landmark haversine columns. Listings with NULL for a signal are dropped from that ranking only — never tail-ranked.
  7. Reciprocal Rank Fusion (RRF, k=60): every ranking contributes 1 / (60 + rank) to each listing's score; results are sorted by the summed score. Fusion is order-only; it doesn't trust raw scores from different models on the same scale.
