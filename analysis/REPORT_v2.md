# Datathon-2026 Corpus — Data Audit **v2 (post-enrichment)**

**Scope.** Every listing in [`data/listings.db`](../data/listings.db) (25,546 rows) plus every derived artifact we built on top of it:

- **Layer 1 — [`listings_enriched`](../enrichment/schema.py)**: 37 null-filled fields × 4 provenance columns each (value / source / confidence / raw snippet).
- **Layer 2 — [`listings_ranking_signals`](../ranking/schema.py)**: 30 derived ranking signals.
- **Layer 2 — [`listing_commute_times`](../ranking/scripts/t4_r5_commute_matrix.py)**: 125,396 real transit-minute pairs.
- **Semantic — [`embeddings.fp16.npy`](../data/ranking/embeddings.fp16.npy)**: 25,546 × 1024 Arctic-Embed-L v2 document vectors.
- **Gazetteer — [`landmarks.json`](../data/ranking/landmarks.json)**: 45 curated Swiss landmarks.

**Provenance.** Every stat in this report was generated by [`analysis/profile_v2.py`](profile_v2.py) from the live DB; machine-readable outputs at [`analysis/data/stats_v2.json`](data/stats_v2.json); 19 plots at [`analysis/plots_v2/`](plots_v2/). Comparisons against v1 ([`REPORT.md`](REPORT.md)) cite [`analysis/data/stats.json`](data/stats.json).

**Baseline for comparison.** The v1 report audited the **raw 4 CSVs** (22,819 rows, mostly-null structured columns). v2 audits the **enriched DB** (25,546 rows; +2,727 new Comparis rows + every row null-filled). Where helpful, v2 tables show the v1 number alongside the v2 number.

## 1. Executive summary

Spot-checks on 20 real addresses match Google Maps transit mode within ±5 min. See §7.

4. **Zero-null invariant holds across all 41 enriched fields** (37 original + 4 pass-2b shipped today). `bathroom_count`, `bathroom_shared`, `has_cellar`, `kitchen_shared` now have real values on 40.5 – 58.2% of listings; the rest are `UNKNOWN` (never `UNKNOWN-pending`). See §9 and the new §5B.

5. **Query answerability** — the ranker can now evaluate: *"canton = ZH"* on 99.7% of listings, *"has balcony"* on 86.1%, *"< 300 m to transit"* on 93.6%, *"≤ 25 min to HB Zurich"* on 23.9% (only 24% of the corpus is close enough to Zurich to bother measuring — that's correct behaviour, not missing data). See §8.

**The one-line takeaway:** *every piece of data ARCHITECTURE.md asked for is now in the DB, plus a real public-transport commute matrix nobody originally planned but which unlocks ~8 of the 31 user-query templates.*

## 9. Hardening — signals we added after the first ranker draft

Two Layer-2 columns exist specifically to keep the ranker safe from pathological inputs. Both documented in [`_context/LAYER2_STATE.md`](../_context/LAYER2_STATE.md) §9.

### 9.1 `nearest_stop_lines_log` — log-transform of transit-hub magnitude

![Lines count vs log](plots_v2/58_lines_count_vs_log.png)

**Problem**: Swiss GTFS reports distinct `route_id` counts per stop. Gare de Cornavin has **~41,000 distinct route_ids** (every direction × variant × time-of-day of every line that serves it counts separately in the feed's encoding). Percentile-normalising a ranker signal on this raw field would let Cornavin-adjacent listings dominate every transit-score on the board.

**Fix**: added `nearest_stop_lines_log = ln(1 + count)`. Log-bounded, stable, reflects the intuition *"more lines → better-served, but diminishing returns"*.

### 9.2 `price_plausibility` — suspect-outlier flag

| bucket | listings |
|---|---:|
| `normal` | 20,145 |
| `suspect` | **74** |
| NULL (no baseline — bucket < 5) | 5,327 |

`suspect` means `|price_delta_pct_canton_rooms| > 3.0` — i.e. price deviates by more than 300% from the (canton × rooms) bucket median. These are typically mis-categorized listings (multi-room rent listed as "1-room", parking spot at CHF 200 listed with rooms > 0, etc.). The ranker excludes them from percentile normalisation and de-weights them in scoring.

## 10. Remaining gaps + known anomalies

### 10.1 Pass 2b — **completed** (see §3B for full analysis)

The 4 extraction-only fields shipped today are **done**: 53,665 DB writes applied, 19,852 cache entries, ~$3 OpenAI spend, 30/30 spot-check clean, parallel auditor verdict SHIP AS IS. Coverage: `bathroom_count` 58.2%, `bathroom_shared` 54.8%, `has_cellar` 40.5%, `kitchen_shared` 56.6%. Zero-pending invariant holds across all **41** enriched fields.

### 10.2 Lake-landmark mid-water geocoding

3 rows in `listing_commute_times` have `travel_min > 90` (above our max_time cap):

```
listing 221168  Oberägeri ZG   → aegerisee  travel_min=105
listing 6105230 Unterägeri ZG  → aegerisee  travel_min=101
listing 6273923 Unterägeri ZG  → aegerisee  travel_min=101
```

Both towns are **on the Ägerisee shoreline**. Root cause: OSM's canonical point for Ägerisee is a centroid **in the middle of the lake** (lat/lng ≈ 47.133, 8.641) — a place r5py's walk network has no edge to. The router eventually finds some absurd multi-leg path that crosses the cap. Same failure mode partially explains the low reachable-counts for `lac` (36), `zugersee` (34), `aegerisee` (11) — listings near the lake succeed to reach, but inland listings fail.

**Mitigation**: ranker should clamp `travel_min > 90` to NULL (one-line SQL). Proper fix is planned P1.2 — sample OSM lake shoreline polygons instead of using centroids.

### 10.3 Corpus-shape ceiling

Fields that will never be at 90%+ coverage because descriptions rarely mention them:

- `agency_name` / `agency_phone` / `agency_email` — only ~2% of listings disclose broker info in the description
- `year_built` — only ~14% of descriptions mention construction year
- `floor` — only ~43% mention floor number
- `distance_*` (5 raw-CSV fields) — roughly 2% of listings in the raw data had these populated; we correctly do NOT fabricate values

These are **text-extraction ceilings**, not pipeline gaps.
